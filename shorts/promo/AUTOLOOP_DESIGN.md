# promo 자가개선 루프 설계서 (200~300편 데이터기반 진화)

> 상태: **설계(미구현)**. 정본 = `promo/README.md`, 학습 상세 = `HOOK_LEARNING.md`. 이 문서는 그 둘을 "닫힌 루프 + 대량 생산"으로 확장하는 로드맵이다.
> 작성 2026-06-18. 대상 채널 = **YouTube Shorts + Instagram/Reels**.
> 원칙: 사실/수학 기준. 추측은 "미확정"으로 표시. 구현 착수 전 사장님 결정이 필요한 항목은 §8에 모음.

---

## 0. 목표를 정확히 정의

사장님 요구를 분해하면 세 개의 *다른* 목표다. 섞으면 안 된다.

| 목표 | 수단 | 성공 지표 |
|---|---|---|
| **도달 극대화** | 같은/유사 영상 재업로드 (신규 소비자는 처음 봄) | 누적 신규 도달 |
| **학습(개선)** | 변주(후킹/스타일/프리셋)를 *다르게* 줘서 비교 | 변주 축별 성과 차이 |
| **운영 효율** | 생성→렌더→업로드→집계 자동화 | 사람 개입 시간/편 |

핵심: **"재사용"과 "개선"은 반대 방향이다.** 재사용은 변수를 고정해 도달만 늘리고, 개선은 변수를 흔들어 데이터를 만든다. 둘 다 하되, 한 영상이 둘을 동시에 하진 못한다. 설계는 "변주로 학습 → 이긴 변주를 재사용으로 양산" 순서로 둘을 연결한다.

## 1. 왜 12편이 끝이 아닌가 / 진짜 병목

볼륨은 병목이 아니다. 스타일 23 × 프리셋 4 × 토픽 N × 후킹 8 = 조합이 사실상 무한이고, MD 자동생성 + `run_promo_batch.bat`로 수백 편 렌더는 쉽다.

병목은 **성과가 되돌아오지 않는다는 것** 하나다. 지금 `_manifest.csv`는 *만든 것*만 기록한다. "성과 안 나오는 거 지우고 다시"가 성립하려면 *결과*가 그 행에 다시 들어와야 한다. 이 폐루프가 전체 설계의 80%다.

## 2. 아키텍처 (5부품)

```
[1 생성기] → review MD N개 (토픽×후킹×스타일×프리셋, 인사이트 가중, 중복회피)
     │
[2 배치렌더] run_promo_batch.bat → out/promo/*.mp4 + _manifest.csv 1행/편   (이미 있음)
     │
[3 업로드 로그] 업로드 시 outfile ↔ platform ↔ video_id 기록 (신규, 사람/반자동)
     │
[4 귀속 잡] YouTube/Meta API로 video_id별 지표 pull → _results.csv 조인  (신규, ads/ 인프라 재사용)
     │
[5 스코어링·가지치기] 변주 축별 성과 집계 → 하위 컷 / 상위 가중 → 다음 세대 [1]로 피드백
```

부품 2는 완성, 1은 쉬움, **3·4·5가 신규 핵심**이다.

## 3. 데이터 모델

### 3.1 manifest (생산 기록) — 현행 유지 + 키 추가
현행 컬럼: `ts·outfile·nn·slug·preset·hook_pattern·hook_text·styles·music_src·music_start·n_facts`.
- 추가 1개: `variant_id` = `slug+preset+styles+music`의 해시(같은 영상 변주를 안정적으로 식별하는 키).

### 3.2 업로드 로그 — **별도 파일 불필요**(관리시트 운영일지 재사용)
관리대장에 이미 `유튜브`·`인스타` **운영 일지 탭**이 있고 각 행에 `날짜·포맷·주제·링크·조회수·좋아요·비고`가 사람 수기로 들어간다. 업로드는 어차피 사람이 하고 거기에 기록하므로, **별도 `_uploads.csv`를 만들지 않는다**.
- 조인 규약: 업로드 시 그 탭의 **비고(운영메모) 칸에 promo `outfile`**(예 `005_bidaemyeon_showcase_1.mp4`, 없으면 `slug`)를 적는다. 이 한 칸이 영상↔변주를 잇는 유일 키다.

### 3.3 results (성과) — `promo/_results.csv` (구현됨, 시드 헤더만)
`pull_date · platform · outfile · slug · preset · url · views · likes`
- 컬럼은 **시트에 실제 있는 값만**(조회수=views, 좋아요=likes). impressions/saves/retention은 운영일지에 없으므로 안 씀(있으면 나중에 추가).
- 채우는 주체 = Claude(또는 사람). §4 참조.

조인: `_results.outfile → _manifest.outfile → (hook_pattern·preset·styles)`. outfile이 유일 조인키(slug는 표기차로 불안정 — 백업용).

## 4. 귀속 — 관리시트 운영일지에서 읽기 (사장님 결정 반영)

성과 출처 = **관리대장 `유튜브`·`인스타` 운영 일지 탭의 조회수·좋아요**(사람 수기). 새 API·OAuth·토큰 **불필요**. 업로드도 사람이 한다.

흐름:
1. 사람: 업로드 → 운영일지 탭에 평소대로 행 기록(링크·조회수·좋아요) + **비고에 promo `outfile`** 기재.
2. Claude: Drive 스냅샷(`PhoneSpot Sheet Snapshots`)에서 두 탭을 읽어 비고에 promo outfile이 있는 행을 추려 `_results.csv`로 떨굼. (= ingestion 단계. PC가 시트를 직접 못 읽으므로 Drive MCP 가진 Claude가 수행. 폴백 수동입력 = `promo_results_add.py`)
3. `promo_score.py` 가 outfile로 manifest와 조인해 축별 집계.

> 미확정/한계: 운영일지 조회수·좋아요는 **사람이 갱신하는 시점에만** 최신. 자동 일배치를 원하면 `ads/` 트랙(Apps Script)이 유튜브/메타 API로 그 탭을 채우게 확장해야 하나, 그건 **광고운영 별도 task**(여기서 안 건드림).

### 4.1 1차 성과 지표 = 시트에 있는 값으로 고정
운영일지에 실재하는 값은 **조회수·좋아요**뿐이다. 그래서:
- **1차 = 조회수(views)** — 도달/훅 강도 프록시.
- **2차 = 참여율(좋아요/조회수)** — 메시지 적합도 프록시.
- 진짜 목표(카톡 전환)는 운영일지엔 없음 → 나중에 litt/UTM→GA4(이미 `GA4_자동` 탭 존재)와 연결하면 전환 귀속까지 확장 가능. 지금은 1·2차로 시작(결과 보며 조정).

## 5. 스코어링·가지치기 + 통계 게이트 (수학)

### 5.1 왜 개별 영상으로는 못 고르나
비율 지표 p의 95% 신뢰구간 반폭 ≈ 1.96·√(p(1−p)/n), n=노출.
- 예) p=5%를 ±1%p로 보려면 **n ≈ 1,800/변주**.
- A안 vs B안에서 2%p 차이(5%→7%)를 80% 검정력으로 잡으려면 **arm당 ≈ 2,000 노출**.
→ 300편을 1편씩 흩뿌리면 개별 영상은 영원히 유의성에 못 닿는다. (사장님 말대로 노출이 작기 때문 — 바로 그래서 개별 비교가 안 됨)

### 5.2 해법: 변주 축으로 풀링
개별 영상이 아니라 **축(hook_pattern / style조합 / preset) 단위로 성과를 합산**한다. "질문형 후킹 50편" 합치면 표본 n이 합쳐져 신호가 빨리 선다. manifest가 이미 이 축을 기록하므로 집계만 짜면 됨. → 후킹 타입을 12편에 채워둔 이유.

### 5.3 단계적 의사결정 규칙
- **데이터 부족기(축당 노출 합 < 게이트, 예 5,000)**: 자동 가중 금지. `유튜브_인사이트`·`메타_인사이트` Top 패턴 휴리스틱으로 생성기 가중(사람/시트 기준).
- **데이터 충분기**: 축별 성과로 ε-greedy 또는 Thompson sampling(다중슬롯) — 이긴 축 비중↑, 탐험 비중 유지. 하위 축 가지치기.
- **가지치기 = 영상 삭제가 아니라 "그 변주를 다음 세대에 덜 뽑기"**. 이미 올린 영상은 도달 자산이니 성과 0 아닌 한 유지(재사용 목적).

⚠ 표본 작을 때 순위는 노이즈다. 게이트 전엔 "이긴 것처럼 보이는 것"을 믿지 않는다 — 이건 통계적 한계이지 구현으로 못 없앤다.

## 6. 재사용 & 플랫폼 정책 (도달 목표 쪽)

- 사장님 논리(노출이 작으면 재업로드해도 신규 소비자엔 처음)는 **도달 관점에서 맞다**.
- 리스크: 거의 동일 영상을 같은 채널에 반복하면 YouTube/Meta의 reused-content·스팸 정책에 걸릴 수 있다. → 변주(스타일·후킹·배경·길이)를 충분히 줘 "다른 영상"으로 만들면 위험↓. 정책은 자주 바뀌므로 **게시 전 현행 정책 확인**(미확정 — 필요 시 검색 확인).
- 운영 규칙(초안): 같은 토픽은 변주를 바꿔 재생산, 동일 파일 그대로 반복 업로드는 피함.
